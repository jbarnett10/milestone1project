Key Concepts

These core ideas appear across HTTP versions and help explain why the protocol could evolve without breaking the way applications think about requests and responses.

  • HTTP Semantics: Stable methods, status codes, headers, and URIs define how clients and servers talk to each other, regardless of which version or transport is underneath.
  • Statelessness: Each request is independent; servers do not automatically remember previous requests. Features like cookies, sessions, and tokens are layered on top to provide continuity.
  • Persistent Connections: HTTP/1.1 reuses a TCP connection for multiple requests, which reduces handshake overhead and improves throughput compared to one-connection-per-request.
  • Multiplexing: HTTP/2 and HTTP/3 allow many streams over a single connection so that multiple resources can be fetched in parallel without opening extra connections.
  • HPACK: The header compression format in HTTP/2 that removes repeated header bytes across requests, shrinking bandwidth usage on chatty sites.
  • QUIC: A UDP-based transport with built-in encryption and stream multiplexing that underpins HTTP/3 and improves performance on lossy and mobile networks.
  • Head-of-Line Blocking: In TCP, a single lost packet can stall later data on the same connection. QUIC reduces this by handling streams independently at the transport layer.
  • Version Independence: Modern specs separate shared semantics (RFC 9110) from individual mappings (HTTP/1.1, HTTP/2, HTTP/3), so applications can think in “HTTP terms” while the wire format evolves.

Together, these concepts show how HTTP kept its core semantics stable while evolving the transport layer from simple one-request-per-connection behavior to multiplexed, encrypted, QUIC-based delivery.
