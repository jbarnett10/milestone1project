HTTP/1.1 and HTTP/2

HTTP/1.1 standardized persistent connections, chunked transfer encoding, virtual hosting via the Host header, and a mature caching model. It became the default across major web servers and browsers for roughly two decades.

HTTP/2 (2015) preserved HTTP semantics but introduced a binary framing layer that multiplexes many streams over a single TCP connection and compresses headers using HPACK. This design reduces handshake overhead and latency compared to the one-stream-per-connection approach in HTTP/1.1. Priority signaling was simplified in later updates, and server push saw limited benefit and is now disabled by default in Chrome.

In practice, multiplexing lets browsers fetch many resources in parallel without opening extra connections, while header compression eliminates repeated metadata. Users see faster page loads and fewer stalls on image-heavy or script-heavy sites, and developers can rely less on workarounds like domain sharding and resource inlining.

Together, HTTP/1.1 and HTTP/2 transformed HTTP from a simple text-based protocol into a more efficient, scalable foundation for the modern web, paving the way for HTTP/3 and QUIC.

Quick Comparison

Key Differences Between HTTP/1.1 and HTTP/2
Feature HTTP/1.1 HTTP/2
Connection Usage Persistent; one active stream per TCP connection Many independent streams over a single TCP connection
Headers Plain text, repeated on each request Binary representation with HPACK compression
Primary Benefit Correctness and practicality at Internet scale Lower latency and better utilization of a single connection
